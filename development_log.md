# Obsidian 小米笔记同步插件 - 开发复盘日志 (Development Log)

本文旨在记录「小米笔记同步插件」从原型到生产级稳健架构过程中的关键设计、异常坑点及优化路径。

## 1. 核心架构演变

### 1.1 隔离沙盒 (Partition Rotation) 架构
- **背景**：早期版本使用固定 `persist:xiaomi` 分区，导致「清除授权」后无法彻底清除 HttpOnly 的服务端 Cookie，用户无法切换账号。
- **决策**：引入**分区轮转机制**。每次用户点击清除，插件直接生成一个新的随机分区 ID（如 `persist:minotesync_123456`），彻底舍弃旧有物理存储。
- **收益**：实现了 100% 的物理级注销，保护了用户隐私。

### 1.2 隐形 Webview 守卫 (CORS & ensureOrigin)
- **发现**：小米云服务前端采用 Vue.js 框架，其心跳重载逻辑常导致隐藏 Webview 偏移小米域，触发 `Failed to fetch` 跨域报错。
- **处理**：在 `MiNoteAPI` 中增加了 `ensureOrigin` 钩子。
- **设计**：每次请求发起前，强制将 Webview 导航回 `https://i.mi.com/favicon.ico` 等静态资源点。通过「锁定静态资源域」规避了官方前端 SPA 路由的干扰。

## 2. 核心功能实现方案

### 2.1 高级模板引擎
- **设计思路**：采用 Handlebars 占位符风格（`{{variable}}`）。
- **实现**：内置 `renderTemplate` 解析器，支持从 note 对象中提取 `id`, `title`, `folder`, `createTime`, `modifyTime` 等变量。
- **防撞名机制**：在文件落地阶段引入 `while` 循环保护。若检测到同名文件异主，自动追加 `_1`, `_2` 序号，确保数据零覆盖风险。

### 2.2 聚合单文件模式 (Aggregate Mode)
- **挑战**：如何在大文件中精准更新单篇笔记而不破坏用户的手动编辑？
- **方案**：**ID 锚点标记法**。
- **逻辑**：使用 `<!-- mi-note-id: xxx -->` 作为起始/闭合锚点。
    - **Update**：正则定位锚点区间，原地替换内容。
    - **Create**：在文件末尾追加。
- **收益**：支持单文件「All in one」流派用户，且实现了极佳的增量更新体验。

## 3. 故障排查与调优记录 (Troubleshooting)

| 故障现象 | 根因分析 | 修复方案 |
| :--- | :--- | :--- |
| 设置面板卡死、空白 | `checkLoginStatus` 阻塞了 UI 主线程 | 引入异步骨架屏，将授权检查逻辑移出生命周期阻塞点 |
| 标题含斜杠导致保存失败 | 文件名中含有非法字符 `\ / : * ? " < > \|` | 构建 `sanitizePath` 强力清洗函数 |

| 登录窗口被拦截/白屏 | Electron 分区隔离未正确注入 Webview | 手动操作 DOM 动态设置 `partition` 属性并强制 reload |

### 2026-03-25: MiNote2Daily 日记同步功能
- **背景**：用户希望将当日的小米笔记快速记录到 Obsidian 日记中，需要灵活的插入位置。
- **优化**：

    - **独立模板**：提供专门的日记插入模板及变量助手。
- **收益**：实现了小米笔记与日记流的无缝集成，增强了碎碎念记录的连续性。

### 2026-04-09: P0 安全与稳定性批量修复
- **背景**：经过全量代码审查，发现 5 项可能导致安全漏洞、内存泄漏或插件挂死的关键缺陷。
- **修复内容**：
    1. **Login 轮询内存泄漏** (`login.ts`)：关闭登录弹窗后，Cookie 检测的 `setTimeout` 递归轮询仍在后台运行。引入 `isDestroyed` 标志位并增加 30 秒超时自动终止机制。
    2. **API URL 注入风险** (`api.ts`)：`executeJavaScript` 中 URL 直接拼接进模板字符串，存在代码注入风险。改用 `JSON.stringify(url)` 安全转义。
    3. **请求超时兜底** (`api.ts`)：`request<T>` 和 `downloadAttachment` 均无超时保护，Webview 卡死时整个插件 hang。引入 `Promise.race` 超时竞争机制（API 请求 30s，附件下载 60s）。同时为 `getProxyWebview` 增加 8 秒创建超时。
    4. **有序列表转换错误** (`utils.ts`)：`<order>` 标签含有 `inputNumber` 属性但被错误转为无序列表。修正为使用 `inputNumber` 生成正确的有序编号。
    5. **批量同步请求节流** (`sync.ts`)：大量笔记同步时无节流控制，可能短时间内向云端发起数百请求触发风控。引入每 20 篇休眠 500ms 的批次节流，并每 50 篇自动持久化同步状态作为断点续传 checkpoint。
- **收益**：消除了全部已知的 P0 级安全/稳定性隐患，显著提升了大规模笔记同步场景下的可靠性。

### 2026-04-09: 解决授权弹窗“秒过场”与静默 SSO 恢复
- **背景**：在 Obsidian 重启后，由于短期 Session(userId) 会被Electron丢弃但主域凭证(pass_token)仍然存活，老的代理探针在纯图片端点上探测不到凭据，导致频发误报让用户点击授权弹窗。点击打开后动态 h5 页面引发 SSO 重定向光速续签，造成“过场闪退”诡异体验。
- **优化**：在 `MiNoteAPI.checkLoginStatus` 中追加“智能唤起流程”。若初检失败，不再直接判死，而是静默驱使代理内核跳转一次 `note/h5`，给予 8 秒宽带捕捉网关底层的 302 重签发回路响应。
- **收益**：彻底消灭多此一举的操作体验缝隙，实现真正意义上的「持久化静默免登和 Token 无感洗白」。
